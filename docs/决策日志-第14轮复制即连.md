# 决策日志-第14轮 复制即连

## 本轮目标
在上一轮“连接串统一载荷”的基础上，再减少一次用户思考与输入，让连接路径尽量接近：

> 复制连接串 → 打开 App → 直接连接

---

## 1. 为什么这轮继续收缩 UI

虽然上一轮已经把主路径从“地址 + 配对码”收成了“连接串”，但页面上仍然同时暴露了：
- 连接串输入
- 配对码输入

这仍然会让用户产生一个潜意识问题：
> 我到底该填哪个？

只要用户会问这个问题，说明连接界面还没有收够。

---

## 2. 本轮做法

### 选择
- 主流程只保留一个输入框：连接信息
- 手动配对码输入移到高级选项
- App 打开页面时自动尝试读取剪贴板中的连接串

### 原因
这是当前阶段成本最低、收益最高的简化：
- 不需要等扫码能力落地
- 不需要新服务端协议
- 不需要改 token / pair 逻辑
- 却能明显减少用户操作

---

## 3. 本轮实际效果

用户现在的连接路径更像：
1. 在服务器侧复制连接串
2. 打开 App 的“添加龙虾”
3. App 自动预填（如果剪贴板里正好有连接串）
4. 点击连接

只有当自动路径失败时，用户才会：
- 手动粘贴连接串
- 展开高级项
- 手填地址 / 配对码

---

## 4. 为什么这是合理的中间态

真正的最终形态大概率是：
- 扫码
- 自动发现
- 一键连接

但在那之前，“复制即连”是最顺的过渡形态，因为它：
- 与二维码使用同一连接载荷
- 与消息/聊天分享天然兼容
- 对现有骨架侵入很小

---

## 5. 当前结论

这一轮不是新功能，而是连接体验打磨：

> 把“粘贴一段信息即可连接”继续推进到“很多时候连粘贴都省了”。

下一步如果继续简化，最值得补的是：
1. 真正的二维码扫描入口
2. 局域网自动发现
3. 安装完成后直接展示二维码
