# 决策日志-第13轮 连接方案简化

## 本轮目标
重新检查当前 Clawboard 连接路径，把用户不该理解的技术细节继续折叠掉，尽量把首次连接收敛成一件事：

> **粘贴一段连接信息即可。**

---

## 1. 现状问题

虽然之前已经把 Bridge 地址放进“高级选项”，但实际链路仍然存在一个问题：
- 用户还是容易被“地址 + 配对码”这两个输入概念影响
- 安装脚本也还在默认输出这两项
- 这意味着底层实现细节还在向用户泄漏

从产品视角看，真正合理的心智应该是：
- 服务器给手机一段连接信息
- App 自动理解它
- 用户只做“粘贴并连接”

---

## 2. 本轮选择

### 方案 A：继续要求用户只输入配对码
**优点**
- UI 比“地址 + 配对码”更简单

**缺点**
- 默认地址策略一旦不成立，还是要用户理解 Bridge 地址
- 不利于后续二维码统一载荷

### 方案 B：引入统一连接串
例如：
`clawboard://pair?code=...&url=http://...`

**优点**
- 一个字符串就能包含连接所需信息
- 后续二维码、剪贴板、消息分享都能复用同一载荷
- 地址可以继续藏起来，不在 UI 主流程里暴露
- 和未来扫码配对天然兼容

**缺点**
- 需要 App 解析这段内容
- 服务端/安装脚本也要一起输出

### 当前选择
选择 **方案 B：统一连接串**。

---

## 3. 本轮实际改动

iOS：
- 新增 `BridgePairingPayload.parse(from:)`
- `PairingFlowView` 主输入改为“粘贴连接信息 / 配对串 / 二维码内容”
- 用户仍可兜底手输配对码
- 高级连接选项保留手填地址，但不再是主路径

Bridge：
- `/pair/session` 增加：
  - `bridge_url`
  - `pairing_link`

安装脚本：
- 默认输出“连接串”
- 只把“地址 + 配对码”保留为调试兜底

文档：
- 更新连接方案说明
- 明确 App 应优先支持“扫码 / 粘贴连接串”

---

## 4. 为什么这是更合理的中间态

因为当前产品还没完全补上二维码扫描，但连接串先行有几个直接好处：
- 不用等扫码能力也能获得近似体验
- 分享路径可以很轻（复制/消息发给自己即可）
- 不会浪费未来扫码工作，二维码内容直接复用连接串

换句话说：
> 连接串是扫码之前最合适的过渡载荷，也是扫码之后仍然保留的底层标准载荷。

---

## 5. 当前结论

这一轮之后，Clawboard 的连接方案更像产品而不是调试面板：
- 用户主流程：粘贴连接串 → 连接
- 高级兜底：手填地址 / 配对码
- 服务端输出：连接串优先

下一步如果继续优化，最自然的是：
1. 真正加入二维码扫描
2. 在安装/启用阶段直接显示二维码
3. 自动发现同局域网 Bridge，进一步减少手动输入
