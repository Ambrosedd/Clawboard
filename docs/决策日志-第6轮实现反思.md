# 决策日志：第 6 轮实现反思

## 本轮目标
把“连接服务器上的龙虾”从工程师路径收成更像真实用户路径的产品入口。

---

## 决策 014：连接页默认先展示什么？
### 备选方案
1. 默认展示 Bridge 地址 + 配对码
2. 默认只展示“输入配对码”，把地址放进高级选项

### 对比
#### 方案 1 优点
- 更贴合当前底层实现
- 调试直接

#### 方案 1 缺点
- 普通用户不理解 Bridge 地址
- 一进页面就暴露工程概念
- 与“添加龙虾”的产品心智冲突

#### 方案 2 优点
- 用户只需要理解“配对码”
- 更接近未来扫码配对
- 地址仅在自建/调试场景出现

#### 方案 2 缺点
- 默认地址需要靠约定或后续自动发现

### 当前选择
选择 **方案 2**。

### 反思
这一步不是改 UI 小细节，而是在收用户心智。当前阶段更需要先让用户知道自己在“添加龙虾”，而不是在“配置本地 API 地址”。

---

## 决策 015：前期 skill 没进官方库时怎么分发？
### 备选方案
1. 只给源码仓库
2. 只给提示词让龙虾自己安装
3. 仓库内 skill + 一键安装脚本 + 极短连接说明

### 对比
#### 方案 1 优点
- 最省实现

#### 方案 1 缺点
- 用户负担大
- 不像产品

#### 方案 2 优点
- 很像 AI-native 产品

#### 方案 2 缺点
- 不可控
- 难稳定交付

#### 方案 3 优点
- 可版本化
- 可审计
- 降低门槛
- 能尽快验证连接路径

#### 方案 3 缺点
- 仍不是最终官方库形态

### 当前选择
选择 **方案 3**。

### 反思
这轮先追求“把连接这一步跑顺”，而不是一次把官方分发体系做满。先有可靠前期方案，再收敛到官方库安装。
