# Clawboard 第三方依赖策略

## 原则
- 优先引入低耦合、高价值库
- 不在产品方向尚未稳定时过早引入重型架构依赖
- 依赖必须服务于交付速度和稳定性，而不是为了“看起来专业”

---

## 推荐优先考虑的依赖

### 1. KeychainAccess
用途：
- 存储 Connector 配对 token
- 替代手写 Keychain 细节代码

优点：
- API 清晰
- 实用价值高
- 侵入性低

缺点：
- 多一个外部依赖

结论：
- 适合在接入真实配对流程时优先引入

---

### 2. swift-log
用途：
- 统一 App 和 Connector 日志结构
- 调试配对、审批、网络请求、事件流

优点：
- 标准化日志抽象
- 后续扩展方便

缺点：
- MVP 阶段收益不如 Keychain 明显

结论：
- 可作为第二批依赖引入

---

### 3. Alamofire（待定）
用途：
- 更成熟的 HTTP 请求组织、拦截器、重试、上传下载

优点：
- 适合复杂网络层

缺点：
- 当前阶段过早引入收益有限
- 原生 URLSession 足够支撑第一轮接入

结论：
- 暂缓，待 Connector 真实接入时再决策

---

## 当前不建议优先引入
- 重型架构框架
- 重型路由库
- 大型 UI 组件全家桶

原因：
- 当前最重要的是尽快跑通产品闭环，而不是把工程抽象做得很“满”。
