# Clawboard 实施计划与决策日志

## 1. 执行约定
从当前开始，项目默认按以下方式推进：

- 按已确认的需求文档实现全部功能点
- 优先保证：**App First / 本地优先 / 低操作复杂度 / 安全边界清晰**
- 遇到实现分歧时，由代理先自主决策
- 每个关键动作必须保留：
  - 目标
  - 可选方案
  - 优缺点对比
  - 选择理由
  - 后续反思

这份文档就是后续的持续决策记录。

---

## 2. 总体实施顺序

### Phase 1：iOS Demo 升级为高保真可运行前端
目标：
- 把现有 SwiftUI 骨架升级成接近真实产品的 Demo
- 覆盖首页、龙虾、任务、审批、设置、配对流程
- 用统一主题和组件系统收敛视觉
- 用 Mock 行为闭环让交互更真实

### Phase 2：Connector 后端骨架
目标：
- 建立轻量本地服务
- 实现需求文档和 API 草案中的基础接口
- 提供 mock 状态、任务、审批、告警与控制命令

### Phase 3：App 接真实 Connector
目标：
- 从 MockData 切换到真实 HTTP / 事件流
- 接入配对 token 存储
- 接入加载态、错误态、重试机制

### Phase 4：闭环打通
目标：
- 完成审批→任务继续
- 控制动作→状态变化
- 配对→节点可见
- 告警→详情跳转与处理

## 3. 第一批关键决策

## 决策 001：iOS 是否重度依赖三方架构库？
### 备选方案
1. 一开始就引入重型架构库/路由库
2. 以 SwiftUI 原生能力为主，少量引入高价值基础库

### 对比
#### 方案 1 优点
- 结构更“完整”
- 某些模块能力开箱即用

#### 方案 1 缺点
- 初期复杂度高
- 项目仍在快速演化，重型抽象容易拖慢改动
- 团队/后续维护者理解成本更高

#### 方案 2 优点
- 更适合当前阶段快速迭代
- 便于先把产品路径跑通
- 更符合 MVP 阶段需求

#### 方案 2 缺点
- 后期可能需要再抽象部分能力
- 某些能力要自己补轻量封装

### 当前选择
选择 **方案 2**：先用 SwiftUI 原生导航 + 自建轻量 ViewModel 结构，再按需引入三方库。

### 反思
这不是反对三方库，而是避免在产品仍高频变化时过早把自己绑进一套重架构里。

---

## 决策 002：网络层是否立刻引入 Alamofire？
### 备选方案
1. 立刻引入 Alamofire
2. 先保留 URLSession / ConnectorClient 轻封装，等接真实接口时再决定

### 对比
#### 方案 1 优点
- 请求封装、拦截器、重试能力成熟
- 后续复杂网络行为扩展较方便

#### 方案 1 缺点
- 当前还处于 Mock 阶段，引入过早收益有限
- Xcode 工程目前刚搭好，先减少不必要变量更稳

#### 方案 2 优点
- 当前阶段足够
- 不影响后续切换
- 更利于先把产品流程打通

#### 方案 2 缺点
- 后续真实接口接入时需要再做一次技术决策

### 当前选择
选择 **方案 2**：先不急着接 Alamofire，等 Connector 接入阶段统一评估。

### 反思
如果后续出现：统一鉴权、请求重试、上传下载、SSE/WebSocket 管理复杂度上升，再引会更合理。

---

## 决策 003：哪些第三方库适合优先引入？
### 当前建议优先级
1. `KeychainAccess`：适合 token 存储
2. `swift-log`：适合统一日志
3. 事件流阶段再评估 SSE / WebSocket 方案

### 当前不急着引入
- 重型状态管理框架
- 重型路由框架
- 大型 UI 全家桶

### 反思
优先引入“低耦合、高价值”的库，而不是“看起来很完整”的库。

---

## 决策 004：Connector 应该作为独立产品暴露给用户吗？
### 备选方案
1. 继续以“独立 Connector”作为主要产品表述
2. 保留 Connector 职责，但将其产品化为“安装在龙虾上的 skill + 本地 bridge/sidecar”

### 对比
#### 方案 1 优点
- 技术分层表达更直接
- 后端/桥接职责容易单独理解

#### 方案 1 缺点
- 用户需要理解额外概念
- 接入门槛偏高
- 与龙虾已有 skill 能力不够贴合

#### 方案 2 优点
- 更符合龙虾生态
- 用户接入路径更自然：安装 skill → 扫码配对 → 在 App 中查看和控制
- 工程上仍然保留本地 bridge/connector 职责边界

#### 方案 2 缺点
- 工程表述需要更精细，避免把 skill、runtime、bridge 混成一层

### 当前选择
选择 **方案 2**。

### 反思
这不是否定 Connector，而是让 Connector 的职责以更产品化的方式嵌入龙虾生态。用户不必理解 Connector，但工程上仍需保留连接、审批桥接、控制、授权边界这些职责。
