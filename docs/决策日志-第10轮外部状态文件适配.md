# 决策日志-第10轮 外部状态文件适配

## 本轮目标
把 Bridge 从“只会读内置 seed 数据”推进到“可以承接外部真实运行态输入”的中间形态。

---

## 1. 为什么这轮不直接硬接真实 runtime IPC

如果现在一步到位做“Bridge 直接深接某个 Lobster Runtime 私有 IPC / 进程协议”，会有几个风险：
- 绑定某个尚未稳定的 runtime 内部实现
- 调试与部署复杂度立刻升高
- 一旦 runtime 结构调整，App/Bridge 侧很容易跟着返工
- 容易把 Bridge 逐渐做成“执行引擎的一部分”，破坏边界

而当前更合理的中间态是：

> **先让 runtime / skill / sidecar 任一采集层，把真实状态整理成 JSON 快照文件；Bridge 负责把这个状态以稳定 API + SSE 的方式暴露给 App。**

这符合当前产品架构：
- 运行态采集 与 App-facing control plane 分离
- Bridge 继续保持窄边界
- App 不感知底层实现细节

---

## 2. 本轮为什么选“外部状态文件”

### 方案 A：继续只用 seed 数据
**优点**
- 最简单
- 演示稳定

**缺点**
- 产品会一直停留在 demo 感
- 无法承接真实服务器状态
- 不利于验证 skill-hosted bridge 的实际价值

### 方案 B：外部状态文件注入
**优点**
- 非常容易和现有脚本/skill 结合
- Bridge API 不用大改
- 可热重载
- 很适合当前阶段验证真实链路

**缺点**
- 还不是最终 runtime adapter
- 文件同步的实时性与原子性需要采集侧保证

### 方案 C：直接 IPC / socket / 私有 SDK 深接 runtime
**优点**
- 更“正统”
- 最终态更可能走向这里

**缺点**
- 当前阶段太重
- 容易过早锁死实现

### 当前选择
选择 **方案 B：外部状态文件注入**。

这是一个高性价比中间层：
- 比 seed 真实很多
- 比深接 runtime 风险小很多

---

## 3. 本轮实际补了什么

Bridge 新增能力：
- `STATE_FILE=/path/to/runtime-state.json`
- 启动时读取 JSON 文件
- 文件变化时自动热重载
- 健康检查返回 `state_source`
- 触发 `runtime.state.reloaded` SSE 事件

同时补充了：
- `connector/sample-runtime-state.json`
- `.env.example` 中新增 `STATE_FILE`
- `connector/README.md` 使用说明
- `install/README.md` 当前接入建议

---

## 4. 这轮带来的实际意义

这让系统从：
- Bridge 自己硬编码一份示例状态

推进到：
- **Bridge 可以消费外部真实状态快照**

于是当前链路已经可以变成：

```text
采集脚本 / runtime adapter / skill sidecar
        ↓ 输出 JSON 快照
      STATE_FILE
        ↓
   Clawboard Bridge
        ↓ API / SSE / Pair / Auth
      iOS App
```

这非常接近未来真正 skill-hosted bridge 的职责分层。

---

## 5. 仍然没做的事

这轮故意没碰：
- 真正的 runtime 私有协议接入
- 状态文件写入原子性保障
- schema 版本协商
- 多 node 聚合写入
- approvals / actions 回写给采集层的反向通道

这些都应该在下一阶段逐步补，而不是这轮一次做满。

---

## 6. 当前结论

这一轮不是最终 runtime adapter，但它把 Clawboard 往“真能挂在现有环境上跑”推进了一大步：

> **Bridge 已经不再只能演示，它现在可以承接外部真实运行态。**

下一步如果继续推进，最值得做的是：
1. 定义状态文件 schema 版本与写入约定
2. 增加一个采集脚本 / adapter 示例，把某个实际 runtime 状态转换为该 JSON
3. 设计控制动作和审批结果如何安全回写给采集层
