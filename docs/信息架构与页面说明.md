# Clawboard 信息架构与页面说明

## 1. 产品结构
Clawboard 当前定位为一个 **iPhone App First、本地优先** 的龙虾管理应用。

用户通过 App 直接连接 Connector，完成四件事：
- 观察龙虾状态
- 查看任务进度
- 处理审批请求
- 在异常时进行干预

---

## 2. 底部导航结构

### 2.1 首页
目标：
- 让用户 3 秒内理解当前整体状态
- 快速看到今日最重要的问题
- 快速进入待处理任务或审批项

展示内容：
- 在线龙虾数
- 运行中任务数
- 待审批数
- 异常数
- 今日重点提醒
- 最近活跃龙虾

### 2.2 龙虾
目标：
- 以“龙虾”为主视角管理运行单元
- 快速识别哪些龙虾正常、哪些卡住、哪些需要处理

展示内容：
- 龙虾列表
- 状态筛选
- 搜索
- 风险标签
- 当前任务摘要
- 所属节点

### 2.3 任务
目标：
- 以“任务”为主视角查看执行进展
- 对复杂任务提供时间线和步骤细节

展示内容：
- 任务列表
- 任务详情
- 进度条
- 时间线
- 当前卡点
- 输入/输出摘要
- 建议动作

### 2.4 审批
目标：
- 集中处理所有敏感动作
- 先突出高风险审批，再处理一般审批

展示内容：
- 待审批项列表
- 风险等级
- 审批原因
- 审批范围
- 时效说明
- 快速批准 / 拒绝 / 缩小范围

### 2.5 设置
目标：
- 降低连接和管理节点的复杂度
- 提供可理解的安全边界

展示内容：
- Connector 配对引导
- 节点列表
- 节点健康状态
- 安全设置
- Token / 配对管理
- 实验功能

## 4. 关键页面说明

## 4.1 首页 Dashboard
首页不是“数据看板”，而是“移动处理台”。

设计重点：
- 第一屏先看待处理事项
- 第二层展示关键计数
- 第三层展示最近活跃与重点异常

用户在首页的核心动作：
- 点击异常进入任务详情
- 点击待审批进入审批中心
- 点击某个龙虾进入龙虾详情

## 4.2 龙虾列表页
龙虾页是用户最常用的管理页面之一。

设计重点：
- 列表卡片清晰展示状态与当前任务
- 风险状态必须醒目
- 节点来源要能看到，便于排查问题
- 搜索与状态筛选要足够轻量

## 4.3 任务详情页
任务详情页是“看进度 + 决策”的核心页面。

设计重点：
- 进度条和当前步骤必须直观
- 时间线帮助理解任务执行上下文
- 当前卡点必须直接给出建议动作
- 控制动作放在底部高可达位置

## 4.4 审批中心页
审批页是整个产品价值最直接的页面之一。

设计重点：
- 高风险审批占据第一视觉层级
- 审批信息要结构化，避免用户阅读成本高
- 默认提供批准/拒绝，还可扩展“缩小范围”等克制动作

## 4.5 设置与连接页
设置页不是传统“配置堆砌”，而是帮助用户建立信任。

设计重点：
- 配对流程要短
- 节点健康可视化
- 安全边界明确：App 不直接持有高危权限，权限在 Connector 本地

---

## 5. 交互原则

### 5.1 单手操作优先
- 底部导航承载一级入口
- 高价值操作尽量靠近底部
- 页面动作避免分散在多个角落

### 5.2 摘要优先，详情下钻
- 列表层只显示最需要判断的信息
- 点击后进入详情页展开上下文

### 5.3 风险优先
- 待审批和异常在视觉上优先级更高
- 风险等级需要颜色和文案双重表达

### 5.4 状态清晰，不堆控制项
- 每个页面只保留最核心动作
- 避免像企业后台那样把复杂操作全部暴露给用户

---

## 6. V2 原型改进总结
相较于上一版原型，V2 的改进包括：
- 从“网页展示感”转为“多页面 iPhone App 感”
- 每个核心页面独立成一张手机画面
- 信息层次更适合后续翻译成 SwiftUI 页面结构
- 审批、任务、连接引导更聚焦用户决策路径
- 视觉上更接近原生产品表达，而不是控制台/落地页
