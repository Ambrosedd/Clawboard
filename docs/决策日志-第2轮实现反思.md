# 决策日志：第 2 轮实现反思

## 本轮目标
把 iOS Demo 从“静态页面集合”推进到“更像真实产品状态机的演示 App”，重点补足加载、空态、错误态，以及任务 / 审批 / 龙虾之间的联动。

---

## 决策 007：错误态是只做一个全局 toast，还是让每个主页面都有可恢复状态？
### 备选方案
1. 继续只在根层显示 toast
2. 为首页、任务、审批、龙虾列表增加明确的 loading / error / empty 视图

### 对比
#### 方案 1 优点
- 改动少
- 实现快

#### 方案 1 缺点
- 用户不知道失败发生在哪个上下文
- 页面会显得“空白但不诚实”
- 不利于后续接真实 Connector 异常

#### 方案 2 优点
- 状态表达更清晰
- 更接近真实 App 的交互预期
- 后续接入真实接口时结构更稳

#### 方案 2 缺点
- 需要统一状态枚举与页面渲染逻辑

### 当前选择
选择 **方案 2**。

### 反思
Demo 阶段最怕“看起来像坏了”。明确的 loading / error / empty 视图，能让演示更可信，也更适合后续替换成真实数据源。

---

## 决策 008：状态联动是继续写死文案，还是把审批和任务建立显式关联？
### 备选方案
1. 保持审批、任务、提醒之间的弱关联
2. 给审批项增加 `taskID` / `lobsterID`，由 ViewModel 统一驱动状态变化

### 对比
#### 方案 1 优点
- 改动轻
- 文件少改

#### 方案 1 缺点
- 审批通过后任务变化不稳定
- 详情页操作容易出现“点了但别的页面没变”
- Demo 真实感不足

#### 方案 2 优点
- 跨页面联动更自然
- 更符合未来真实接口返回结构
- 能支撑暂停、恢复、拒绝、终止等操作闭环

#### 方案 2 缺点
- 需要重构一部分 Mock 数据和 ViewModel

### 当前选择
选择 **方案 2**。

### 反思
这个项目不是一组孤立页面，而是“同一个 agent 工作流在多个页面的投影”。所以显式关联是必要的，不然 Demo 很容易露馅。

---

## 决策 009：要不要为了演示效果引入更复杂的本地存储或状态管理框架？
### 备选方案
1. 提前引入额外状态管理 / 持久化方案
2. 继续保持原生 SwiftUI + 轻量 ViewModel，只把演示状态结构整理清楚

### 对比
#### 方案 1 优点
- 似乎更“工程化”
- 未来持久化可能更方便

#### 方案 1 缺点
- 当前需求还没到那一步
- 会明显增加工程复杂度
- 与“先做轻量 App First Demo”的方向不一致

#### 方案 2 优点
- 保持演示工程干净
- 方便后续再替换成真实 Connector 数据
- 更符合当前避免重依赖的策略

#### 方案 2 缺点
- 某些长期能力（本地缓存、离线恢复）要留到后续补

### 当前选择
选择 **方案 2**。

### 反思
这轮的重点不是把技术栈做重，而是让产品行为做真。先把状态表达、联动逻辑和组件一致性做好，收益比提前上复杂基础设施更高。

---

## 本轮实现摘要
- 增加统一 `AppLoadPhase` 与 Demo 场景切换
- 把 ConnectorClient 改成按场景返回快照 / 抛出错误
- 为首页、龙虾、任务、审批列表补上 loading / error / empty 状态
- 把审批、任务、龙虾、提醒通过 ID 建立更明确的联动
- 让任务详情和龙虾详情都能反映最新状态，而不是停留在初始快照
- 抽出更统一的轻量状态组件，减少重复 UI 写法

## 下一步建议
1. 开始补一个更像真实接口的 Connector mock 协议层（而不是直接 ViewModel 调静态数据）
2. 增加本地缓存与最近一次连接节点记忆
3. 为异常任务补更细一点的诊断信息与重试动作
