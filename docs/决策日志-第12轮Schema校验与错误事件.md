# 决策日志-第12轮 Schema 校验与错误事件

## 本轮目标
把 Bridge 的 `STATE_FILE` 接入从“格式约定存在”推进到“运行时可校验、可观测、出错不静默”。

---

## 1. 为什么这轮优先做错误可见性

在已有 schema 和 adapter 示例之后，最大的风险已经不是“没人知道怎么接”，而是：

> **接入方写错了，但 Bridge 只是默默失败或悄悄读出异常内容。**

这种情况对产品伤害很大：
- App 看见的数据可能莫名不更新
- 现场排查困难
- 接入方会误以为是网络、配对、App、或 bridge 控制逻辑坏了

所以这轮最值钱的事情是：
- 校验输入
- 保留上一份有效状态
- 明确暴露错误

---

## 2. 为什么选择“degraded + 保留最后有效状态”

### 方案 A：状态文件非法时直接崩溃退出
**优点**
- 非常严格
- 问题暴露直接

**缺点**
- 生产环境体验差
- 一个坏写入就让整个 Bridge 掉线
- App 侧会感知为节点失联，而不是状态文件格式错误

### 方案 B：状态文件非法时继续吃进去
**优点**
- 表面上不断服务

**缺点**
- 会污染状态
- 风险比崩溃更坏，因为是假正确

### 方案 C：保留最后有效状态 + 健康降级 + 发错误事件
**优点**
- 服务仍可用
- 错误显式可见
- 不污染现有状态
- 更适合真实生产桥接环境

**缺点**
- 需要额外维护状态来源状态机

### 当前选择
选择 **C**。

这很符合 Bridge 的定位：
> 它是本地控制面，不应该因为一次坏写入就瞬间失明，但也不能假装一切正常。

---

## 3. 本轮实际补了什么

Bridge 新增：
- 轻量内建 schema 校验器
- `stateSourceStatus` 运行态
- `/health` 返回 `degraded` 与详细 `state_status`
- `/device/info` 补充 `state_status`
- SSE 事件：`runtime.state.invalid`

行为变更：
1. `STATE_FILE` 加载前先校验
2. 校验通过：替换状态并发 `runtime.state.reloaded`
3. 校验失败：保留上一份有效状态，记录错误并发 `runtime.state.invalid`

---

## 4. 为什么先用内建校验器而不是引第三方 schema 库

当前目标不是做完整 JSON Schema 引擎，而是先解决：
- 顶层字段有没有
- 时间格式像不像样
- 关键数组和关键字段齐不齐
- 出错时能不能把信息给出来

在这个阶段，引完整 schema validator 的收益不如保持实现轻：
- 无额外依赖
- 方便 sidecar / 本地部署
- 容易读懂和继续改

如果后续 schema 更复杂，再考虑接真正 validator 也不迟。

---

## 5. 验证结果

已实际验证：
1. 用有效 state 文件启动 Bridge
2. 建立 SSE 订阅
3. 将 live state 文件替换成非法结构
4. 观察结果：
   - `/health` 变成 `degraded`
   - `state_status.last_error` 可读
   - SSE 收到 `runtime.state.invalid`
   - 上一份有效状态仍保留

这说明系统已经具备“接错能报、报了不死、还能继续服务”的能力。

---

## 6. 当前结论

这一轮让 Clawboard Bridge 从：
- 有 schema 文档
- 有 adapter 示例

进一步推进到：
- **运行时真的会校验 schema，并把错误明确暴露出来**

这是让外部接入进入“可维护”阶段的关键一步。

下一步最值得做的是：
1. 给 invalid state 增加更明确的运维提示 / docs
2. 在 iOS 端显示节点健康降级状态
3. 设计控制动作与审批结果如何安全回写给 adapter / runtime
