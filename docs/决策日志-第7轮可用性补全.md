# 决策日志-第7轮可用性补全

## 本轮目标
把 Clawboard 从“已经能配对和展示真实数据”继续推进到“核心操作路径基本可用”。

当前判断：
产品离可用不远，主要缺的不是新页面，而是 **真实控制闭环**。

---

## 1. 我对当前缺口的判断

在本轮之前，系统已经具备：
- Bridge 配对
- token 签发与撤销
- App 读取真实 `/lobsters` `/tasks` `/approvals` `/alerts`
- iOS 侧连接信息持久化

但还存在一个关键问题：

> App 虽然能连上真实 Bridge，但很多操作按钮仍主要在改本地演示状态，没有把“审批 / 暂停 / 恢复 / 终止 / 重试”完整走到真实服务端。

这会导致三个问题：
1. 用户以为自己在控制真实龙虾，实际上只是本地 UI 在变化。
2. 刷新后状态可能回跳，前后不一致。
3. 产品已经跨过“演示期”，这类假闭环会严重伤害信任感。

所以这一轮最重要的不是继续堆功能，而是把已有按钮改成真的。

---

## 2. 本轮选择补哪些

### 选择补的
1. iOS `ConnectorClient` 增加真实控制调用：
   - approve
   - reject
   - pause
   - resume
   - terminate
   - retry
2. `AppViewModel` 在存在 Bridge 连接时：
   - 优先走真实接口
   - 成功后立即 `refresh()` 重新拉取服务端状态
   - 失败时向用户显示明确 toast
3. 任务详情 / 龙虾详情页按钮改为异步调用
4. 为失败/终止任务补上 `retry` 路径
5. 设置页增加“当前已启用真实 Bridge 拉取与控制闭环”说明

### 暂时不补的
1. SSE / WebSocket 事件流
2. 真正的 iOS Keychain 落地
3. 多设备 token 管理界面
4. QR 扫码输入体验
5. 更深的 runtime adapter 真实对接

---

## 3. 为什么这样排优先级

### 方案 A：先补事件流
**优点**
- 更像实时产品
- 用户感知更高级

**缺点**
- 如果控制按钮还是假的，实时流也只是把“看上去在线”做得更像
- 会把复杂度拉高，但不先解决信任闭环

### 方案 B：先补真实控制闭环
**优点**
- 直接提升“可用”而不是“像可用”
- 让 App 与 Bridge 之间的契约更真实
- 后续加事件流时，基础语义已经对齐

**缺点**
- 用户感知没有事件流那么显眼
- 仍然需要 refresh 拉取，不是推送式

### 当前选择
选择 **方案 B：先补真实控制闭环**。

原因很直接：
> 一个管理产品，先要“真能控”，再要“控得很丝滑”。

---

## 4. 这轮完成后的可用性提升

完成后，真实 Bridge 连接模式下：
- 批准 / 拒绝审批：会直接调用服务端审批接口
- 暂停 / 恢复 / 终止任务：会直接调用服务端控制接口
- 失败或已终止任务：可以直接重试
- 每次操作成功后：都会重新拉取快照，避免 UI 与服务端状态漂移
- 操作失败时：用户会得到明确错误反馈，而不是静默失败

这意味着当前版本已经更接近一个“可用 MVP”：
- 能连
- 能看
- 能批
- 能控
- 能重试

---

## 5. 仍然存在的缺口

虽然这一轮把闭环补上了，但距离“顺手好用”还差这些：

### A. 连接体验
- 还主要靠手输地址/配对码
- 二维码扫描与自动发现未完成

### B. 安全落地
- 凭证边界已抽象，但仍未切到真正 Keychain
- 多 token 管理 / 当前会话查看还没进入 App UI

### C. 状态同步体验
- 现在主要靠操作后 refresh
- 缺少 `/stream/events` 之类的增量同步

### D. Runtime 真实接入深度
- 当前 server 仍偏示例/骨架
- 要真正监控“现有 lobsters running on servers”，还需要更真实的 runtime adapter

---

## 6. 我对“产品可用”的当前判断

如果只按 MVP 标准：
- **演示可用：已经达到**
- **早期真实试用：开始接近可用**
- **可交给外部非技术用户稳定使用：还不够**

最主要的原因不是前端不够完整，而是：
- 服务端仍是 bridge skeleton
- 真实 runtime adapter 还不够深
- 连接方式还不够傻瓜

所以接下来最应该继续补的是：
1. runtime adapter 真实化
2. 事件流 / 增量同步
3. 扫码与连接体验
4. Keychain 与 token 管理完善

---

## 7. 本轮结论

本轮不追求“看起来更完整”，而是优先把：

> **真实 Bridge 模式下的控制操作变成真正可信的闭环。**

这是从 Demo 迈向产品最关键的一步之一。
