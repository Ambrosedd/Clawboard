# 决策日志：第 5 轮实现反思

## 本轮目标
从“文档上定义 skill-hosted bridge”推进到“App 能开始真实配对 Bridge、仓库里有正式 skill 设计、并完成一轮白盒测试记录”。

---

## 决策 011：今晚优先做什么，才算真正推进功能性开发？
### 备选方案
1. 继续打磨纯 Demo 页面的表现
2. 优先补真实接入链路最小闭环：pair 协议、iOS 配对调用、skill 设计、白盒测试

### 对比
#### 方案 1 优点
- 视觉上更完整
- 交互感继续增强

#### 方案 1 缺点
- 对真实接入帮助有限
- 会继续停留在“演示产品”阶段

#### 方案 2 优点
- 开始把 App 从 Demo 推向真实接入
- 让 skill、bridge、iOS 三者开始在协议上收敛
- 更贴近当前产品阶段目标

#### 方案 2 缺点
- 视觉收益没那么直接
- 会暴露更多工程未完成项（比如 Keychain、真机编译、runtime adapter）

### 当前选择
选择 **方案 2**。

### 反思
这轮如果还继续只磨 Demo 外观，等于回避真正的难点。当前更重要的是打通最小接入链路。

---

## 决策 012：skill 要不要只放在工作区，不进入主仓库？
### 备选方案
1. skill 仅保留在工作区 `skills/`
2. skill 同步进入 `Clawboard` 主仓库版本管理

### 对比
#### 方案 1 优点
- 对本地 agent 使用方便
- 不影响主仓库目录结构

#### 方案 1 缺点
- 用户要求的 GitHub 版本管理无法覆盖这部分成果
- 设计与项目主线分离，后续容易漂移

#### 方案 2 优点
- skill 设计与项目主线一起版本化
- 文档、bridge、iOS 可以一起演进
- 更符合“做好版本管理”的要求

#### 方案 2 缺点
- 仓库内容更多，需要更注意结构整洁

### 当前选择
选择 **方案 2**。

### 反思
这轮中途发现这个问题是好事，及时纠正比事后补救强。

---

## 决策 013：当前环境没有 Swift 编译器时，怎么做白盒测试？
### 备选方案
1. 等真机/Xcode环境后再说
2. 先做可执行的 bridge API 白盒测试 + skill 校验 + iOS 静态逻辑检查，并明确记录限制

### 对比
#### 方案 1 优点
- 不会产生“不完整测试”的心理负担

#### 方案 1 缺点
- 今晚这轮实际上没有测试交付
- 无法证明关键协议骨架真的可运行

#### 方案 2 优点
- 至少能对白盒测试给出真实执行证据
- 能发现 skill/YAML 这种真实错误
- 能让“已完成”和“未完成”边界更清楚

#### 方案 2 缺点
- 不能替代最终的 Xcode 编译验证

### 当前选择
选择 **方案 2**。

### 反思
这轮实践证明这个选择是对的：skill frontmatter 的 YAML 问题就是在校验时暴露出来的，不测就会带着假完成状态进入仓库。
